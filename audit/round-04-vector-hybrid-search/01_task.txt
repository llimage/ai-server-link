三、实现要求：LanceDB 接入
==================================================

请完成 LanceDB 真实接入，作为 knowledge chunk 向量存储层。

要求：

1. 建立向量库适配层
- 不允许业务代码直接操作 LanceDB SDK
- 通过独立 adapter / repository / store 封装

2. 向量存储对象至少包含：
- chunkId
- documentId
- sourceId
- tenantId（如现有结构已有）
- text
- embedding
- metadata

3. 向量表初始化
- 启动时或首次写入时可自动初始化
- 不要求复杂迁移系统，但需要稳定可重复执行

4. 向量删除/覆盖策略
- 对同一 chunk 重复 index 时应具备幂等性
- 不允许无限重复脏写

==================================================
四、实现要求：Embedding Provider
==================================================

请实现 embedding provider 抽象层。

要求：

1. provider 抽象
- embedding 调用必须通过 provider adapter
- 不允许 search/ingestion 直接依赖某个具体实现

2. 第一批实现
- 先提供一个“本地可运行”的 embedding provider
- 可基于 deterministic mock-real embedding（固定维度，稳定可测试）
- 输出必须稳定、可复现，不允许随机漂移影响测试

3. profile / config
- embedding dimension、batch size、provider name 可配置
- 与 deployment profile 保持兼容思路

==================================================
五、实现要求：Ingestion 与向量化
==================================================

请把 ingestion pipeline 的 chunk/index 阶段与向量化联通。

流程仍为：
1. upload
2. parse
3. chunk
4. index
5. publish

新增要求：

1. chunk 阶段
- 照常将 chunk 写入数据库 knowledge_chunks

2. index 阶段
- 对 chunk 生成 embedding
- 写入 LanceDB
- 将 index 状态写回数据库 / ingestion 状态记录

3. publish 阶段
- publish 后才允许进入正式 hybrid 检索结果
- 未 publish 内容不可进入正式结果

4. 失败处理
- embedding / vector write 失败时应将任务标记为 failed
- 错误信息记录到 ingestion_logs 或 metadata

6. 兼容性
