七、建议实现边界（可在此范围内设计，但不能越界）
==================================================

你可以在不改架构前提下，补齐以下能力：

1. reminder domain model
建议至少包含：
- id
- tenantId
- userId
- planId（可为空，若由计划触发）
- type
- channel
- scheduledAt
- status（pending / due / running / sent / failed / cancelled）
- payload
- dedupeKey
- retryCount
- lastError
- createdAt
- updatedAt
- sentAt / executedAt

2. schedule 计算器
需要支持至少：
- 单次提醒
- 每日提醒
- 漏打卡后升级提醒
- 面向亲属的提醒升级链路

3. reminder queue
要求：
- 至少支持“待执行任务入队”
- 可按时间取到期任务
- 有基本幂等/去重能力
- 不能重复无限执行同一任务

4. dispatcher / worker
要求：
- 能拉取 due 任务
- 能执行并写入结果
- 失败可记录
- 具备最小重试策略（若你实现，必须可审计；若不实现，必须明确说明）

5. notifier abstraction
要求：
- 先实现系统内部通知抽象层
- 可以先用“持久化发送记录 + 可观测日志”作为链路完成标准
- 不要求真实外部厂商
- 但接口必须为未来短信/推送接入保留合理位置
- 不能先把真实第三方 SDK 硬编码进业务逻辑
